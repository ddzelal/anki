# CLAUDE.md

Project conventions for the Anki project. Read this before working with Google Sheets or Supabase.

## Design system (match ulum.rs)

UI follows the Ulum Academy look: **light theme**, **Roboto** font, rounded cards, soft shadows, generous spacing. Palette is defined as Tailwind 4 `@theme` colors in `app/globals.css` — use these utilities, don't hardcode hex:

- `ulum-blue` `#284b70` (primary / headings / front of card) · `ulum-blue-dark` for hover
- `ulum-green` `#77a290` (success / answers / progress bar) · `ulum-green-dark` for hover
- `ulum-pink` `#cc3366` (accent / "Again" / errors)
- `ulum-yellow` `#fdd05a` (accent / "Hard", use dark text on it)
- `ulum-cream` `#f4efe6`, `ulum-paper` `#faf7f2` (backgrounds), `ulum-ink` `#33373d` (text)

Pages: `/` landing, `/study` (flashcards + name + progress card with stats: Ukupno/Naučeno/Za danas/Nove), `/admin` (cache-refresh + group diagnostics — **no sync**, cards are read live from the Sheet). The logged-in user's name + avatar initial show in the `/study` header.

## Architecture (READ FIRST — changed 2026-07-13)

**The Google Sheet is the single live source for cards; the DB stores only per-user progress.** There is **no sync step and no cards table** anymore. On each `/study` load, `/api/cards/due` reads `CardsV2` + `Settings` live from the Sheet (cached ~5 min in memory per serverless instance; `?fresh=1` bypasses), filters visible cards by the student's Moodle groups **in JS**, and joins against the user's progress rows. A card is identified by a **stable `card_key = base64url(sha256(front + back))[:22]`** (`lib/cardkey.ts`), so progress survives without any card table. Editing a card's `front`/`back` in the Sheet gives it a new key (its old progress is orphaned — rare, acceptable for a vocab deck). This replaced the old "pull from Sheet → sync into `anki_cards`/`anki_card_groups`" design, which was the source of the group-filtering bugs.

## Supabase

- **Project:** `apwwevsyealtsnywwzak` (shared — we hit the Supabase project limit; also hosts a courts/booking app).
- **Our app tables:** in `public`, prefixed `anki_` (visible in Table Editor). Only **three** — cards/groups/settings live in the Sheet now, not the DB:
  - `anki_users` (moodle_sub, display_name)
  - `anki_reviews` (FSRS state per user/card, keyed by `(user_id, card_key)`; `card_key` = hash of `front+back` from `lib/cardkey.ts`; `introduced_at` = when a card was first seen, used for the daily new-card limit). **Stores the full ts-fsrs `Card`, incl. `learning_steps`** (the position in the learning steps `1m`/`10m`). This column is REQUIRED: without it a card can't graduate Learning→Review and stays stuck at 10 min forever — dropping any ts-fsrs Card field silently breaks scheduling. Migration `scripts/add-learning-steps.ts`.
  - `anki_review_log` (one row per rating event, keyed by `(user_id, card_key)`; powers the day-streak)
  - All have **RLS enabled with no policies** → not reachable via the public PostgREST API. The server talks to them via direct Postgres connection (`DATABASE_URL` pooler), which bypasses RLS.
  - **Removed 2026-07-13:** `anki_cards`, `anki_card_groups`, `anki_settings` (migration `scripts/migrate-to-cardkey.ts` backfilled `card_key` from `anki_cards` then dropped them). Do NOT recreate — cards/groups/settings are Sheet-only.
- **ltijs system tables:** live in `public` (unprefixed: `platforms`, `idtokens`, `contexttokens`, `platformStatuses`, `publickeys`, `privatekeys`, `accesstokens`, `nonces`, `states`, `SequelizeMeta`). Schema isolation was attempted (`anki_lti`) but is **not feasible**: ltijs-sequelize migrations use unqualified table names and the transaction pooler (required for serverless) ignores the `search_path` startup option. Names don't collide with the courts app. All are **RLS-locked** (they hold private keys/tokens; ltijs connects as `postgres` which bypasses RLS).
- For OUR app tables, never create bare-named tables in `public` — always `anki_` prefix. (ltijs tables are the documented exception, managed by the library.)
- **Card data source of truth = the Google Sheet, read live** (no DB copy). Cards never enter Postgres; only per-user progress does. Do NOT hand-write card data — manual transcription of Arabic corrupts rows.
- Access via the `supabase` MCP server (needs a Personal Access Token) or directly via the `DATABASE_URL` pooler (one-off scripts use `./node_modules/.bin/tsx --env-file=.env.local`).

## Moodle

- Instance: `https://learn.ulum.rs` (`NEXT_PUBLIC_MOODLE_URL`). LTI 1.3 platform, registered (client `HwtO0SEjOBUdFAO`, deployment `1`). Launch flow: `/api/lti/login` (OIDC) → `/api/lti/launch` → sets `anki_session` cookie → 303 redirect to `/study` (no ltik in the URL).
- **Identity via a signed session cookie** (NOT ltik). At launch, `onConnect` builds `{sub, name, groups}`, signs it with `LTI_KEY` (jsonwebtoken, 30d) and sets the httpOnly cookie `anki_session` (`lib/session.ts`), then 303-redirects to `/study` (no ltik in the URL). The App Router routes `GET /api/cards/due` + `POST /api/review` read that cookie via `resolveUser` (`lib/identity.ts`): valid cookie → real student (`token.user` = Moodle `sub` → `anki_users`, own FSRS progress, name shown in `/study`); no/invalid cookie → dev fallback (`dev-local`, sees all). This makes refresh work and decouples from ltik expiry. The student's group IDs are read once at launch from the LTI **custom parameter** and cached in the cookie.
- **Access model = per-word groups, Model B, filtered live in JS.** A card is shown to a student **iff it is checked (`TRUE`) for at least one group the student is in**; a card with no group checked is shown to **nobody**. `filterVisible(cards, groupIds)` in `lib/sheets.ts`: `groupIds === null` (GROUPS_ENABLED off, OR **admin**, OR dev) → sees ALL; otherwise keeps only cards whose `groups[]` intersects the student's. Admin/dev bypass is in `/api/cards/due` (`GROUPS_ENABLED && !isDev && !isAdmin ? groups : null`) so a teacher in no group still previews everything. `lesson` is display metadata only. **Consequence: a real student in NO group (or whose groups have no checked words) sees NOTHING — every student must be in a group with checked words.**
- **Group access = LTI custom parameter, NOT NRPS.** Core Moodle does **not** send group names via NRPS (groups aren't in the [IMS NRPS spec](https://www.imsglobal.org/spec/lti-nrps/v2p0); only the contributed celtic-project plugin adds them). What core Moodle *can* do is the custom-parameter substitution `$Moodle.Person.userGroupIds`, which yields the **numeric group IDs** the user is in (no names). So: in the Moodle External Tool config add a Custom parameter **`groupids=$Moodle.Person.userGroupIds`**. At launch `getGroupsFromCustom` (`lib/lti.ts`) reads `token.platformContext.custom.groupids`, splits on `,`/`;`/space → `groups` = ID strings, cached in the session cookie. We match by ID (not name) because Moodle gives no name — the sheet header carries both: `Name (ID)`, display name for humans, ID for matching.
- **⚠️ Why `groupids` comes back EMPTY (the recurring bug): the substitution is empty for a user who is in NO group** — which is exactly the instructor/admin case. It is *not* a wiring bug. Always test group filtering by launching as a **student who is a member of a group**, never as yourself/instructor. Other empty causes: launch not in a course context; custom param not actually saved on the tool (each line `key=value`, no quotes); very old Moodle. The `$Context.id` substitution (course id) and the `sub` claim (Moodle user id) are always present. Confirmed via Moodle docs/tracker (MDL-72421): `userGroupIds` is IDs-only and empty for non-members.
- **✅ ACTIVE mechanism = Moodle Web Services (implemented 2026-07-13).** On this Moodle (5.1.3) the custom-param `$Moodle.Person.userGroupIds` substitutes to **empty** even when the student IS in a group (confirmed via the full-`custom`-claim dump on `/admin`: `custom.groupids` came back `""`, likely because the group is in no *grouping*). So at launch `onConnect` (`lib/lti.ts`) calls `fetchUserCourseGroups(courseId, sub)` (`lib/moodle.ts`) → REST `core_group_get_course_user_groups` → returns each group's **id AND name**. `courseId` is derived from the token with **no extra custom param** (`courseIdFromToken`: parses `/CourseSection/<id>/` out of the NRPS `context_memberships_url`, which is present when NRPS is on). The student's cached `groups` = **both** the numeric ids and the names, so a Sheet header can be `Ime` OR `Ime (ID)` — both match `filterVisible`. The custom-param path is kept as a **fallback** if `MOODLE_WS_TOKEN` is unset or WS returns nothing.
  - **Setup (once, admin):** enable Web Services + REST protocol; External services → add function `core_group_get_course_user_groups` → Create token for a user with `moodle/site:accessallgroups` (admin/manager); set `MOODLE_WS_TOKEN` on Vercel. No Sheet change needed — headers are already group names.
  - `sub` (LTI `token.user`) = the Moodle **numeric user id**, which is what the WS `userid` needs.
- **Flags:** `GROUPS_ENABLED` (default false) gates **filtering** — false → `filterVisible(cards, null)` (everyone sees every active card); true → `filterVisible(cards, user.groups)` (in `/api/cards/due`). `GROUPS_DEBUG=true` is a safe **diagnostic** that does NOT filter: `/api/me` returns `{groups, groupsRaw, groupsDebug}` and `/admin` shows the raw `groupids` custom value + parsed IDs so the teacher can confirm wiring and discover the IDs. `RESOLVE_GROUPS = GROUPS_ENABLED || GROUPS_DEBUG` gates whether the custom param is read at all. `[GROUPS_DEBUG]` console logs (Vercel runtime logs) print the custom claim + parsed ids.
- **Rollout:** (1) deploy; (2) add Custom param `groupids=$Moodle.Person.userGroupIds` in Moodle + set `GROUPS_DEBUG=true` on Vercel; (3) find each group's numeric ID (Moodle: Course → Participants → Groups → select group → the URL `.../group/index.php?...&group=ID`, or **launch as a student in that group** and read the ID on `/admin`); (4) put `Name (ID)` in the CardsV2 group headers (`pnpm checkboxes` keeps the checkbox widgets working regardless of header text); (5) tick the words in the Sheet (no sync — live); (6) set `GROUPS_ENABLED=true`, drop `GROUPS_DEBUG`. Known groups: `arapski_jezik_decembar_2025`, `Šapat Kur'ana` (IDs TBD — read on /admin as a student).
- **Discarded approach (history):** first tried NRPS with `?groups=true` (`getMembers`) — doesn't work on core Moodle (returns members but no group data), confirmed empty on a live `GROUPS_DEBUG` launch. Replaced by the custom-param approach above.
- `pnpm checkboxes` (`scripts/setup-group-checkboxes.ts`) uses the service account with read-write scope to set BOOLEAN data-validation (checkboxes) on every group column (E+); MCP can't set data validation, this script can.
- **Admin = LTI role.** At launch, `onConnect` checks the roles claim; Instructor/Administrator/Manager → `isAdmin` in the session cookie. `/api/me` exposes `{name, isAdmin}`. In `/study` the header menu (≡) shows admin actions (Admin panel) only for admins; everyone gets "Osveži iz Sheet-a" (busts the 5-min cache via `?fresh=1`) / Home. `/admin` has a manual "Osveži iz Sheet-a" + the group diagnostics. Dev fallback (no cookie) is admin only in development (`NODE_ENV !== 'production'`), never in prod. (There is no `/api/sync` / `SYNC_SECRET` flow anymore.)
- The Next.js↔ltijs bridge (`pages/api/lti/[...path].ts`) must `delete req.cookies` (Next pre-populates it, which makes cookie-parser skip and never set `req.secret` → signed-cookie failure).
